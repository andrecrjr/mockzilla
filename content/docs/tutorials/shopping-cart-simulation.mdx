# Tutorial: Creating a Dynamic Shopping Cart

In this tutorial, we move beyond simple variables and use the **Mini-DB**. This is a persistent internal data store that allows your mocks to "remember" items in a list—perfect for shopping carts, to-do lists, or message feeds.

## What You'll Learn
- How to create a **Mini-DB Table**.
- How to "push" items into a table with `db.push`.
- How to list table contents in your response.
- How to clear state with `db.remove`.

---

## Step 1: Scenario Setup
1.  Navigate to **Workflows** in the dashboard.
2.  Click **Create Scenario** and name it `my-store`.
3.  Your base URL: `http://localhost:36666/api/workflow/exec/my-store/`

---

## Step 2: Add an Item (POST /cart)
We want to "capture" the body of a POST request and save it into a table called `cart_items`.

1.  Click **Add Transition**.
2.  **Name**: `Add to Cart`
3.  **Path**: `/cart`
4.  **Method**: `POST`
5.  **Effects**: Add a `db.push` effect.
    - **Table**: `cart_items`
    - **Value**: `{{input.body}}`
6.  **Response**:
    ```json
    {
      "message": "Item added!",
      "items_count": "{{db.cart_items.length}}",
      "session_id": "{{state.id}}"
    }
    ```

---

## Step 3: List the Cart (GET /cart)
Now we need to retrieve everything stored in the `cart_items` table.

1.  Click **Add Transition**.
2.  **Name**: `List Items`
3.  **Path**: `/cart`
4.  **Method**: `GET`
5.  **Response**:
    ```json
    {
      "cart": {
        "items": {{db.cart_items}},
        "total_items": "{{db.cart_items.length}}"
      }
    }
    ```

---

## Step 4: Conditional Discounts (Advanced)
Let's add logic that returns a discount code only if the user has more than 2 items in their cart. This demonstrates **Condition Interpolation** and **Arithmetic**.

1.  Click **Add Transition**.
2.  **Name**: `Get Discount`
3.  **Path**: `/cart/discount`
4.  **Method**: `GET`
5.  **Condition**:
    - **Selector**: `db.cart_items.length`
    - **Operator**: `gt`
    - **Value**: `2`
6.  **Response**:
    ```json
    {
      "eligible": true,
      "code": "LOYALTY_10",
      "items_needed_for_next_tier": "{{5 - db.cart_items.length}}"
    }
    ```

---

## Step 5: Empty the Cart (DELETE /cart)
Finally, let's add a way to clear the table.

1.  Click **Add Transition**.
2.  **Name**: `Clear Cart`
3.  **Path**: `/cart`
4.  **Method**: `DELETE`
5.  **Effects**: Add a `db.remove` effect.
    - **Table**: `cart_items`
    - **Condition**: (Leave empty to clear all items).
6.  **Response**:
    - **Status Code**: `204` (No Content)

---

## Step 5: Test the Flow

### 1. Add some coffee beans
```bash
curl -X POST http://localhost:36666/api/workflow/exec/my-store/cart \
  -H "Content-Type: application/json" \
  -d '{"id": "p1", "name": "Dark Roast Coffee", "price": 14.99}'
```

### 2. Add an espresso machine
```bash
curl -X POST http://localhost:36666/api/workflow/exec/my-store/cart \
  -H "Content-Type: application/json" \
  -d '{"id": "p2", "name": "Precision Espresso Machine", "price": 499.00}'
```

### 3. See your cart content
```bash
curl http://localhost:36666/api/workflow/exec/my-store/cart
```
**Result**: You'll see both items stored correctly in the array!

### 4. Clear the cart
```bash
curl -X DELETE http://localhost:36666/api/workflow/exec/my-store/cart
```

## Why this is better than static mocks
Traditional mocking requires a separate file for an empty cart, a cart with one item, a cart with two items, etc. This is fragile and tedious to maintain.

With **Workflows**, your mock becomes a **true prototype**. Your frontend can add, remove, and list items dynamically, allowing you to test:
- "Empty State" UI components.
- Cart badges and counters.
- List rendering and layout shifts.
