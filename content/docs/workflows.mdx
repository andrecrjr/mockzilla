# Stateful Workflows

Go beyond static mocks. Workflow Mode lets you build smart API simulations that remember previous requests. This is perfect for testing complex user journeys like a shopping cart, a multi-step checkout, or a full authentication flow.

<Alert>
  <AlertTitle>Static Mocks vs. Workflows</AlertTitle>
  <AlertDescription>
    Static mocks always return the same data. Workflows use a **mini-database** and **state variables** to return different responses based on what happened in previous steps.
  </AlertDescription>
</Alert>

---

## 🧠 The "Action-Driven" Philosophy

In Mockzilla, we follow a simple rule: **State variables must not be endpoints.**

In a real app, you rarely "set the database status to 'paid'" via a direct API call. Instead, you "process a payment," and the *result* is that the status becomes 'paid'. Mockzilla workflows mirror this:

- ❌ **Bad:** Creating a `POST /set-paid` endpoint to manually update a variable.
- ✅ **Good:** Creating a `POST /checkout` transition that internally updates multiple state variables and DB tables as a side effect of the action.

---

## 🏗️ Core Concepts

### 1. Scenarios
A **Scenario** is an isolated sandbox. Each scenario has its own independent state and mini-DB. You can have a "Happy Path" scenario and a "Payment Failed" scenario running side-by-side.

### 2. Transitions
A **Transition** is a rule that defines how Mockzilla should react to a request. It consists of:
- **Path & Method:** e.g., `POST /orders/:id`.
- **Conditions:** Does this rule apply to the current request?
- **Effects:** What changes should happen to the state or DB?
- **Response:** What should be returned to the client?

### 3. State & Mini-DB
- **State (`state`)**: A simple key-value store for variables like `isLoggedIn` or `currentUser`.
- **Mini-DB (`db`)**: A collection of JSON tables (arrays) for storing lists of items like `orders`, `products`, or `cart_items`.

## 🔗 Path Matching

Transitions support both exact and parameterized paths:

| Path | Example Match | Accessing Parameters |
|------|---------------|----------------------|
| `/users` | `/users` | (None) |
| `/users/:id` | `/users/42` | `{{input.params.id}}` |
| `/cart/:sku/add` | `/cart/SKU-123/add` | `{{input.params.sku}}` |

## 🛠️ Logic & Control Flow

### Conditions
Conditions determine if a transition fires. If multiple transitions match a path, Mockzilla uses the first one whose conditions are met.

| Type | Description | Example |
|------|-------------|---------|
| `eq` | Equals | `input.body.role` == `"admin"` |
| `neq` | Not equals | `state.status` != `"locked"` |
| `exists` | Field exists | `input.headers.authorization` exists |
| `gt` / `lt` | Greater/Less than | `input.body.quantity` > `0` |
| `contains` | Array/String contains | `db.roles` contains `"editor"` |

### Effects
Effects modify your data when a transition is triggered.

| Type | Description |
|------|-------------|
| `state.set` | Update state variables (e.g., `isLoggedIn: true`). |
| `db.push` | Add a new item to a table (e.g., add to `cart`). |
| `db.update` | Find rows by a `match` field (e.g., `id`) and `set` new values. |
| `db.remove` | Delete all rows from a table that `match` a criteria. |

## 🚀 Example: Shopping Cart Flow

Let's build a flow where a user adds an item to their cart and then views it.

### Step 1: Add to Cart
- **Path:** `POST /cart`
- **Effect:** `db.push` to table `cart` with value `{{input.body}}`.
- **Response:** `{"success": true, "message": "Item added"}`

```bash
# Add item
curl -X POST http://localhost:3000/api/workflow/exec/my-store/cart \
  -H "Content-Type: application/json" \
  -d '{"id": 1, "name": "Coffee Beans", "price": 15}'
```

### Step 2: List Cart
- **Path:** `GET /cart`
- **Response:** `{{db.cart}}` (Returns the array of items pushed in Step 1).

```bash
# Get cart items
curl http://localhost:3000/api/workflow/exec/my-store/cart
# Returns: [{"id": 1, "name": "Coffee Beans", "price": 15}]
```

### Step 3: Checkout (Conditional)
- **Path:** `POST /checkout`
- **Condition:** `exists` on `db.cart[0]` (Only fires if cart is not empty).
- **Effect:** `db.remove` all from `cart` AND `state.set` `orderComplete: true`.
- **Response:** `{"status": "Success", "orderId": "ORD-123"}`

```bash
# Try to checkout
curl -X POST http://localhost:3000/api/workflow/exec/my-store/checkout
```

## 🔌 Using your Workflow

To trigger your workflow, use the specialized execution endpoint:

```bash
# POST to: /api/workflow/exec/{scenario-slug}/{path}
curl -X POST http://localhost:3000/api/workflow/exec/my-store/cart \
  -H "Content-Type: application/json" \
  -d '{"id": 1, "name": "Coffee Beans"}'
```

### State Inspector
While testing, open the **State Inspector** in the Mockzilla Dashboard. It provides a real-time view of:
- All **State Variables** currently set.
- All **Database Tables** and their rows.
- A **Reset** button to wipe the data and start over.

> **Pro Tip:** You can manually edit the state or DB rows in the inspector to test edge cases (like a massive cart or a specific user ID) without having to trigger dozens of API calls.
