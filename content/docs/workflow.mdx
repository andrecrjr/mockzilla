# Workflow Mode

Go beyond static mocks. Build stateful, intelligent API simulations.

## Why Workflow Mode?

> **Philosophy: Action-Driven State**
>
> - **Don't CRUD State** — Creating endpoints just to set data. Ex: `POST /set-data`
> - **Do Business Logic** — Transitions trigger state changes. Ex: `POST /checkout` updates DB.

### Static Mocks vs Workflow Mode

**Static Mocks (The Old Way):**
- Always returns the same JSON.
- No memory of previous requests.
- Impossible to test logical flows (e.g. Empty Cart vs Full Cart).

**Workflow Mode (The New Way):**
- **Stateful:** Remembers logic variables (isLoggedIn, cartCount).
- **Dynamic:** CRUD operations on an in-memory database.
- **Smart:** Responses change based on input or state.

## Core Concepts

| Concept | Description |
|---------|-------------|
| **Scenario** | A named container (e.g. "auth-flow"). It holds all your state and variables in isolation. |
| **Transition** | A rule: "WHEN a request matches X, THEN do Y and return Z". |
| **Mini-DB** | A temporary database for each scenario. Push items, query them, and update them. |

## Deep Dive: Transitions & Conditions

A transition links an incoming request to effects and a response.

### Condition Builder

**Selectors:**
- `input.body.field`
- `input.query.param`
- `input.headers.auth`
- `input.params.id`
- `input.body.items[0].id`

**Operators:**
- `==`, `!=`
- `>`, `<`, `>=`, `<=`
- `exists`
- `contains`

> **Pro Tip: Shorthand Access** — You can omit `input.body` for top-level body fields. `amount > 100` is the same as `input.body.amount > 100`.

## Dynamic Responses & Interpolation

Inject data into your response using `{{ var }}` syntax.

| Syntax | Description |
|--------|-------------|
| `{{ input.body.id }}` | Echo request data |
| `{{ input.body.items[0] }}` | Access array item |
| `{{ state.token }}` | Return stored variable |
| `{{ db.users }}` | Return full table |
| `{{ db.cart.length }}` | Table count |

## Example 1: Complex Business Logic (Action-Driven)

**POST /checkout**

```jsonc
// 1. Create Order in DB
{ "type": "db.push", "table": "orders", "value": { "id": "ord_123", "items": "{{ state.cart }}" } }

// 2. Clear Shopping Cart State
{ "type": "state.set", "raw": { "cart": [], "cartCount": 0 } }

// 3. Update User Status
{ "type": "state.set", "raw": { "lastOrderDate": "2025-04-10T..." } }
```

## Example 2: Standard CRUD

### Create — POST /users

```jsonc
// Effect: Store user in DB
{ "type": "db.push", "table": "users", "value": "{{ input.body }}" }

// Response
{ "id": "{{ input.body.id }}", "status": "created" }
```

### Read — GET /users

```
// Response: Return all users
{{ db.users }}
```

### Update — PUT /users/:id

```json
{
  "type": "db.update",
  "table": "users",
  "match": { "id": "{{ input.params.id }}" },
  "set": "{{ input.body }}"
}
```

### Delete — DELETE /users/:id

```json
{
  "type": "db.remove",
  "table": "users",
  "match": { "id": "{{ input.params.id }}" }
}
```

## Debugging Tips

- Use the **Inspect State** tool in the MCP server to see what's currently in your properties.
- If a condition isn't triggering, check if you are using the correct type (string vs number) in your comparison.
- Remember that `input.params` comes from the URL path (e.g. `:id`), while `input.query` comes from `?id=...`.
