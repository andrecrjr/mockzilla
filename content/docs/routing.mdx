# Routing & Matching

Mockzilla's routing engine gives you full control over how your incoming requests match your mocks. This means you can create multiple responses for the same path based on URL segments or query parameters.

## Path Matching Strategies

When you create or edit a mock, look for the **Match Type** dropdown in the **Advanced Options** tab.

| Strategy | How it Works | When to Use It |
|----------|--------------|----------------|
| **Exact** | The path must match exactly what you typed. | Static endpoints like `/login` or `/profile`. |
| **Wildcard** | Use `*` to capture dynamic URL segments. | Paths with IDs like `/users/*` or `/api/*/orders`. |
| **Substring** | Matches if the path is contained anywhere in the URL. | Prefixes like `/api/v1` that catch everything under that route. |

### Wildcard Variants

With a **Wildcard Match Type**, you can set up **Variants** to return different data for different captured segments.

For example, on `/users/*`:
- If the ID is `1`, create a variant with key `1` to return a "Gold Member" profile.
- If the ID is `2`, create a variant with key `2` to return a "Basic" profile.
- You can add a fallback `*` to catch any other IDs with a default response.

### Multiple Wildcards

Mockzilla supports multiple wildcards in a single path. When you use more than one `*`, the **Variant Key** is formed by joining each captured segment with a **pipe (`|`)** character.

**Example Pattern:** `/api/users/*/orders/*`

| Request Path | Captured Segments | Variant Key to Use |
|--------------|-------------------|--------------------|
| `/api/users/123/orders/456` | `123`, `456` | `123|456` |
| `/api/users/alice/orders/latest` | `alice`, `latest` | `alice|latest` |

To handle these in the mock editor:
1. Set the Match Type to **Wildcard**.
2. Add a variant.
3. In the **Key** field, enter the joined values (e.g., `123|456`).

## Matching by Query Parameters

You can also enforce strict matching based on the **Query String** (the part after the `?` in the URL). This is great for testing pagination or feature flags.

In the **Query Parameters** section of the mock editor:
1. Add a **Key** (like `page`) and a **Value** (like `1`).
2. Mockzilla will only serve this mock if the incoming request matches those exact parameters.
3. You can stack multiple parameters to create highly specific test cases.

---

## Strategy Precedence

Mockzilla follows a logical order to find the best match:
1. **Exact** matches are checked first (they are the most specific).
2. **Wildcard** matches are checked next.
3. **Substring** matches are checked last (they are the most general).

> **Pro Tip:** Specificity is your friend. If you have a `/users/admin` mock (Exact) and a `/users/*` mock (Wildcard), Mockzilla will correctly pick the `admin` version first.
